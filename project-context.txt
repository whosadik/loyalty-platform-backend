название проекта - Üilesim

1) Идея проекта

Development of a Loyalty System Platform for a Chain of Cosmetic Stores Based on Personalized Recommendations.

Платформа объединяет персонализацию и лояльность: система понимает предпочтения клиента (кожа/волосы/макияж/ароматы), рекомендует товары, и выдаёт следующий лучший оффер (скидка/мультипликатор баллов) с учетом ограничений бюджета и правил. Все действия измеримы через аналитику (CTR/CR, redemption, promo efficiency, retention proxy).

2) Основные функции и фичи
A) Каталог и карточка товара (реальный ритейл)

Хранение товаров 4 категорий: skincare/haircare/makeup/fragrance

Внутри категории: product_type (cleanser, shampoo, lipstick, edp…)

Расширенный контент карточки:

изображения (image_url, image_urls)

описание (description), применение (application_text)

состав (ingredients_inci)

Импорт каталога из Excel/CSV, поддержка source_product_id (чтобы маппить внешние данные).

B) Профиль пользователя (4 профиля в одном)

Пользовательский профиль хранит предпочтения:

skincare: skin_type, goals, avoid_flags, budget

haircare: hair_profile (JSON: тип волос, scalp, проблемы)

makeup: makeup_profile (finish, coverage, undertone и т.д.)

fragrance: fragrance_profile (семейства, ноты, интенсивность)

Профиль используется как контекст для рекомендаций и для выбора оффера.

C) Лояльность (баллы и уровни)

LoyaltyAccount: points_balance, tier

Tier: Bronze/Silver/Gold (порог по spend за 90 дней, points_rate)

Ledger (EARN/REDEEM): прозрачная история начислений/списаний

Готовые бизнес-фичи:

bonus за заполнение профиля (one-time, idempotent)

начисление баллов за покупки (автоматически через checkout)

списание баллов (если включено) при покупке

D) Персональные рекомендации товаров

Три ключевых сценария:

For You — персональные рекомендации по профилю и поведению

Because you bought X — сопутствующие товары (“покупают вместе”)

Trending — топ продаж за период (как fallback и мерчандайзинг)

Поддерживаются фильтры: category/product_type/price range, in_stock.

E) Next Best Reward (персональный оффер)

Система выдаёт следующий лучший оффер:

discount (скидка) или points_multiplier (множитель баллов)

таргетинг оффера:

cart (на весь чек)

category

product_type

product_id

ограничения:

cooldown_days, expires_in_days

anti-repeat (не давать один и тот же product_id X дней)

saturation (не предлагать то, чего у пользователя и так много)

category cooldown

Ключевая фишка: кампании (Campaigns) с бюджетом и приоритетом:

weekly_limit / weekly_spent (weekly reset)

allowed_categories / allowed_steps

routing кампаний по контексту (например fragrance cross-sell после покупки fragrance).

F) Полный lifecycle офферов + аналитика

События офферов фиксируются в OfferEvent:

assigned / exposed / clicked / redeemed / expired

идемпотентность по event_key/request_id

Это позволяет честно считать:

redemption rate по exposed

CTR clicks/exposed

эффективность промо (promo_efficiency proxy)

breakdown по кампаниям

G) Аналитика рекомендаций + A/B тесты

События рекомендаций: RecommendationEvent:

impression / click / add_to_cart / purchase_attributed

пишется rank, algo_used, experiment_id/variant

A/B эксперименты:

sticky bucket по user_id (control=cooc, test=reranker)

guardrail: авто-откат на baseline при деградации CR

отчёты:

by_algo (cooc vs reranker)

by_experiment (experiment_id/variant)

админ эндпоинты для сводки и фильтрации

4) Логика системы и ключевые алгоритмы
4.1 Алгоритм checkout (атомарный поток)

Цель: провести покупку консистентно и прозрачно.

Шаги:

принять список items (product_id, quantity), channel

посчитать totals:

subtotal

eligible_total (что подпадает под оффер)

если apply_assignment_id:

проверить assignment (не expired, не redeemed)

применить скидку/мультипликатор через общий pricing модуль

если redeem_points:

проверить баланс

списать через ledger (REDEEM)

создать Transaction + TransactionItem

обновить OwnedProduct (пользователь “владеет” купленным)

начислить баллы через ledger (EARN) с учетом tier/мультипликатора

отметить оффер redeemed (и записать OfferEvent.redeemed)

сформировать post_ctx (categories/product_types/product_ids) и авто-назначить next offer

вернуть ответ

Особенности:

идемпотентность покупки по idempotency_key (replay-safe)

один pricing модуль используется в checkout + preview

4.2 Рекомендации: кандидатный слой + reranker
Candidate generation (baseline)

co-occurrence из покупок (что покупают вместе)

multi-context: учитывает последние K покупок (context_k=3)

fallback:

global popularity

category-aware popularity fallback

фильтры: in_stock, исключение owned и т.д.

Reranker (ML)

reranker учится на задаче “следующая покупка”

принимает кандидатов и признаки:

transition_count, rank_inv, same_category, same_brand, price_diff, log_popularity, …

выдаёт пересортировку кандидатов

доступен в API:

algo=cooc|reranker|auto

fallback если модель не загружена

4.3 Next Best Reward: выбор оффера + выбор таргета
Выбор кампании (routing)

hard preference order по контексту (пример):

fragrance purchase → fragrance_crosssell

missing step (например spf) → skincare_retention

иначе default/makeup_push

затем сортировка по priority, проверка gates allowed_categories/steps

Выбор оффера внутри кампаний

Скоринг зависит от:

RFM сегмента (vip/at_risk/new_or_rare)

контекста (steps, post_ctx)

эффективности категории (global effectiveness)

бюджета (weekly_left, stretch)

Выбор таргета

Порядок стратегий:

shortcut по контексту (missing step)

bundle-driven product_id (по rec_bundle)

post-purchase cross-sell rules (шампунь→кондиционер и т.д.)

fallback по профилю/recommend()

Ограничения:

anti-repeat (product_id)

saturation limits по product_type

category cooldown

4.4 Событийный контур и метрики
RecommendationEvent

impression/click/purchase_attributed

хранит rank, algo_used, experiment_id/variant

позволяет считать CTR/CR и сравнивать варианты

OfferEvent

assigned/exposed/clicked/redeemed/expired

позволяет считать чистую воронку офферов (redemption по exposed)

6) Что делает проект “полноценной платформой”

интегрирует лояльность и рекомендации в один цикл,

управляет промо-бюджетом через кампании и ограничения,

поддерживает explainability (“почему дали этот оффер/этот товар”),

имеет событийную аналитику + A/B + guardrails,

обеспечивает консистентность транзакций и auditability.
